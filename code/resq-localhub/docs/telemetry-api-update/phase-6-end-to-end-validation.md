## 9. Scoring Isolation Evidence

Implemented isolation points:

- Backend `MqttSubscriberService` classifies `SENSOR_STREAM` before session scoring and records diagnostics snapshot.
- `TelemetryPayloadNormalizer` identifies firmware live shape but `SENSOR_STREAM` is routed out before scoring.
- Frontend direct MQTT/session normalizer rejects payloads that contain `telemetry_mode`.
- Existing backend tests include `sensorStreamTelemetryUpdatesLiveRegistryButSkipsSessionScoring`.
- Existing frontend tests include `liveClientSensorStreamIsolation.test.ts`.

Required explicit accumulator regression across all summary fields is partially covered, but a full accumulator-baseline test with injected `SENSOR_STREAM` followed by normal session telemetry should be added before release.

## 11. Performance Observations

Estimated payload size from canonical simulator packet is approximately 430-520 bytes depending numeric precision.

| Interval | Packets/sec/device | Approx MQTT payload traffic/device |
|---|---:|---:|
| 100 ms | 10 | about 4-6 KB/s before MQTT/TCP overhead |
| 200 ms | 5 | about 2-3 KB/s before MQTT/TCP overhead |
| 1000 ms | 1 | about 0.5 KB/s before MQTT/TCP overhead |

Observed code risks:

- Backend currently logs `Processed SENSOR_STREAM telemetry for {device}` at INFO for every packet. At 100 ms this can create log noise and should be reduced to DEBUG or sampled before pilot.
- Backend `SensorStreamService` stores one latest snapshot per device and uses per-device SSE emitter lists; no growing snapshot collection was observed.
- Frontend renders latest state as events arrive; if 100 ms causes visible render lag, throttle presentation only while preserving latest received snapshot and backend `lastSeen`.
- Simulator logs every publish unless `--quiet` is used.

## 15. Known Limitations

- Physical hardware validation has not been executed.
- Full broker/backend/simulator smoke was prepared but not executed in this pass.
- Duplicate `request_id` behavior is safe/idempotent at the task level but not backed by a persistent firmware dedupe cache.
- Backend restart behavior while firmware is already streaming needs external validation.
- Frontend reconnect behavior should be tested against a real dropped SSE connection.
- Explicit all-field scoring accumulator regression should be added before release.

## 16. Release Recommendation

Classification: **READY FOR HARDWARE VALIDATION**

Justification: contract surfaces match after simulator invalid-command reason alignment, automated unit/integration coverage exists across backend and frontend, and a deterministic hardware/simulator validation script is available. The feature should not be classified as ready for release until physical device checks and external broker/backend/simulator smoke pass.

## 17. Rollback Plan

Firmware rollback:

- disable `cmd/telemetry` subscription/handler or revert the manual stream phase
- ensure manual stream task cannot start
- preserve existing session telemetry

Backend rollback:

- disable telemetry START/STOP routes
- preserve `SENSOR_STREAM` early-return scoring isolation while any firmware capable of publishing `SENSOR_STREAM` remains deployed
- never roll back the scoring guard before firmware rollback

Frontend rollback:

- hide/remove diagnostics controls
- preserve session isolation logic in direct MQTT/session normalizers

Important: never roll back backend `SENSOR_STREAM` scoring isolation while firmware capable of publishing `SENSOR_STREAM` remains deployed.
