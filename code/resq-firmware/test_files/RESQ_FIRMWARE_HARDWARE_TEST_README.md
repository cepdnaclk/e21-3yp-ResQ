# ResQ Firmware Hardware Test Guide — Updated Firmware Version

This README matches the latest firmware design discussed for ResQ:

```text
BOOT
→ CONFIG_CHECK
→ PROVISIONING
→ WIFI_CONNECTING
→ BACKEND_REGISTERING
→ MQTT_CONNECTING
→ PAIRED_IDLE
→ CALIBRATING
→ READY_FOR_SESSION / CALIBRATION_FAIL
→ SESSION_ACTIVE
→ READY_FOR_SESSION
```

It also includes the newer system-control behavior:

```text
BUTTON_1 long press 3s in normal states → TURN_OFF
BUTTON_2 long press 3s in normal states → RESETTING → BOOT → PROVISIONING
```

`ERROR` and `CALIBRATION_FAIL` own the buttons themselves, so the global long-press actions must not override those recovery states.

## Main firmware contract updates covered

### 1. Calibration payloads are now minimal

Calibration progress topic:

```text
resq/{deviceId}/events/calibration/progress
```

Minimal progress payload:

```json
{
  "event_type": "calibration_progress",
  "reason_id": 204,
  "state": "CALIBRATION_FAIL",
  "action_id": 3,
  "ts_ms": 123456
}
```

Calibration result topic:

```text
resq/{deviceId}/events/calibration/result
```

Minimal result payload:

```json
{
  "event_type": "calibration_result",
  "command_id": "cmd-003",
  "status": "NACK",
  "result": "FAIL",
  "reason_id": 204,
  "state": "CALIBRATION_FAIL",
  "action_id": 3,
  "ts_ms": 123456
}
```

These fields must **not** appear in calibration progress/result payloads:

```text
device_id
reason_code
message
action_code
readyForSession
step
```

LocalHub must map:

```text
reason_id → UI message
action_id → UI instruction
```

### 2. Firmware error payloads are minimal

Firmware error topic:

```text
resq/{deviceId}/events/error
```

Minimal payload:

```json
{
  "event_type": "firmware_error",
  "reason_id": 1400,
  "state": "ERROR",
  "action_id": 6,
  "ts_ms": 123456
}
```

These fields must **not** appear in `firmware_error` payloads:

```text
device_id
reason_code
message
action_code
```

### 3. Active session is now implemented

Expected session flow:

```text
READY_FOR_SESSION
→ cmd/session/start
→ SESSION_ACTIVE
→ telemetry starts
→ cmd/session/stop
→ READY_FOR_SESSION
```

Session telemetry topic:

```text
resq/{deviceId}/telemetry
```

Expected telemetry shape:

```json
{
  "event_type": "session_telemetry",
  "session_id": "S-001",
  "state": "SESSION_ACTIVE",
  "depth_progress": 0.92,
  "depth_ok": true,
  "rate_cpm": 108.0,
  "compression_count": 18,
  "valid_compression_count": 15,
  "recoil_ok_count": 14,
  "incomplete_recoil_count": 3,
  "pause_s": 0.2,
  "hand_placement": "CENTER",
  "pressure_balance_pct": 8.5,
  "flags": "DEPTH_OK,RATE_OK,RECOIL_OK,HAND_CONTACT",
  "ts_ms": 123456
}
```

### 4. MQTT host/port are not stored in NVS

Firmware should store only persistent local config:

```text
wifi_ssid
wifi_pass
backend_base_url
valid calibration config
```

Firmware must **not** store:

```text
mqtt_host
mqtt_port
mqtt_username
mqtt_password
```

On boot, the firmware should call backend registration again and receive MQTT details at runtime.

## Calibration test

Use this when sensors are connected and you are ready to manually apply pressure/compression.

```powershell
  --wifi-ssid "YOUR_WIFI_SSID" `

  --edge-tests `
  --run-calibration `
  --calibration-timeout 240 `

```

The updated firmware does not send `step` names in the calibration MQTT payload. So the test validates the minimal numeric payload contract instead of checking progress step strings.

Run this after the device has reached `READY_FOR_SESSION`.

```powershell
python .\resq_firmware_state_test_updated.py `
  --wifi-ssid "YOUR_WIFI_SSID" `
  --wifi-pass "YOUR_WIFI_PASSWORD" `

  --topic-style short `

```

Expected:

```text

→ ACK or status SESSION_ACTIVE
```


## Manual long-press tests

These cannot be fully automated from Python because they require physical button presses.

| Test | Manual action | Expected result |
|---|---|---|
| Button 1 normal-state turn off | Hold `BUTTON_1` for 3 seconds in `PAIRED_IDLE` or `READY_FOR_SESSION` | State becomes `TURN_OFF`; NVS config is preserved. |
| Button 2 factory reset | Hold `BUTTON_2` for 3 seconds in a normal state | State becomes `RESETTING`; config is cleared; reboot enters `PROVISIONING`. |
| Button 1 in `CALIBRATION_FAIL` | Hold/press according to fail-state behavior | Retry calibration, not turn off. |
| Button 2 in `CALIBRATION_FAIL` | Hold/press according to fail-state behavior | Return to paired idle, not factory reset unless explicitly designed. |
| Button 1 in `ERROR` | Press according to error-state behavior | Retry/recover based on last error. |
| Button 2 in `ERROR` | Press according to error-state behavior | Flush/provisioning behavior. |

## Manual MQTT commands

Replace `<deviceId>` with your actual device ID.

### Debug

```powershell
mosquitto_pub -h 192.168.8.187 -p 1883 -t "resq/<deviceId>/cmd/debug" -m "{`"command_id`":`"cmd-debug`",`"event_type`":`"debug_req`",`"issued_at_ms`":51000}"
```

Expected reply:

```text
resq/<deviceId>/debug
```

### Start calibration

```powershell
mosquitto_pub -h 192.168.8.187 -p 1883 -t "resq/<deviceId>/cmd/calibration/start" -m "{`"command_id`":`"cmd-cal`",`"event_type`":`"calibration_start`",`"hall_delta`":13500,`"ref_pressure`":20100,`"bladder_1_pressure`":15000,`"bladder_2_pressure`":15000,`"issued_at_ms`":51000}"
```

Expected:

```text
resq/<deviceId>/events/calibration/progress
resq/<deviceId>/events/calibration/result
resq/<deviceId>/status
```

### Invalid calibration payload

```powershell
mosquitto_pub -h 192.168.8.187 -p 1883 -t "resq/<deviceId>/cmd/calibration/start" -m "{`"command_id`":`"cmd-invalid`",`"event_type`":`"calibration_start`",`"issued_at_ms`":51000}"
```

Expected minimal payload:

```json
{
  "event_type": "calibration_progress",
  "reason_id": 100,
  "state": "PAIRED_IDLE",
  "action_id": 1,
  "ts_ms": 123456
}
```

### Start session

```powershell
mosquitto_pub -h 192.168.8.187 -p 1883 -t "resq/<deviceId>/cmd/session/start" -m "{`"command_id`":`"cmd-session-start`",`"event_type`":`"session_start`",`"session_id`":`"S-TEST-001`",`"profile_id`":`"adult-basic`",`"issued_at_ms`":51000}"
```

Expected if ready:

```text
SESSION_ACTIVE
telemetry starts
```

Expected if not ready:

```text
NACK calibration_not_ready
```

### Stop session

```powershell
mosquitto_pub -h 192.168.8.187 -p 1883 -t "resq/<deviceId>/cmd/session/stop" -m "{`"command_id`":`"cmd-session-stop`",`"event_type`":`"session_stop`",`"session_id`":`"S-TEST-001`",`"issued_at_ms`":91000}"
```

Expected:

```text
READY_FOR_SESSION
telemetry stops
```

## How to interpret results

| Status | Meaning |
|---|---|
| `PASS` | Expected behavior observed. |
| `WARN` | Behavior was partially observed or depends on hardware/manual action. |
| `FAIL` | Firmware violated the expected state/topic/payload contract. |
| `SKIP` | Test intentionally skipped by command-line options. |

## Recommended validation order

1. Build firmware:

```powershell
idf.py build
```

2. Flash manually if needed:

```powershell
idf.py -p COM4 -b 115200 flash monitor
```

3. Run safe basic test:

```powershell
python .\resq_firmware_state_test_updated.py `
  --host-ip 192.168.8.187 `
  --wifi-ssid "YOUR_WIFI" `
  --wifi-pass "YOUR_PASSWORD" `
  --topic-style short `
  --edge-tests
```

4. Run calibration test.

5. Run active session test.

6. Run system command test last.

7. Run physical long-press tests manually.
